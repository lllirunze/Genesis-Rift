# 《开天辟地之大地的裂变》技术选型与开发阶段规划

## 文档说明

这份文档用于沉淀《开天辟地之大地的裂变》当前阶段适合采用的技术路线。

它服务于两个目标：

1. 确认这不是一个 `3D` 游戏项目
2. 确认我们应当优先把它做成一个以 Web 前端和 TypeScript 为核心的平面桌游式项目

## 一、项目形态判断

### 1. 不做 `3D`

- 这个项目不需要走 `3D` 客户端路线。
- 我们不需要 Unity 风格的立体世界，也不需要实时 `3D` 漫游。
- 这会让项目复杂度过早膨胀，也会偏离“桌游操作系统”的核心目标。

### 2. 目标形态是平面电子桌游

《开天辟地之大地的裂变》最终更像是：

- 桌游规则引擎
- 卡牌界面
- 地图棋盘
- 多个可插拔小游戏模块

### 3. 视觉参考方向

- 整体形态更接近“大富翁式平面棋盘 + 卡牌面板 + 规则结算界面”
- 而不是开放世界 RPG、横版动作游戏或 `3D` 沙盒

### 4. 当前最重要的结论

- 先把它当成“可交互的电子桌游”
- 再逐步提升表现层
- 不要一开始就把它当成传统游戏引擎项目

## 二、开发工具建议

### 1. 优先使用 VS Code

当前阶段推荐使用 `VS Code`。

原因很明确：

- 原生适合 JavaScript、TypeScript、HTML、CSS
- 对 React 与前端工程支持成熟
- 对 Markdown 文档写作友好
- 对 JSON 配置和数据文件处理方便
- 适合一个“代码、文档、配置、资源混合管理”的仓库

### 2. 为什么它适合本项目

这个项目未来会同时包含：

- 游戏设计文档
- React 页面
- TypeScript 规则代码
- JSON 卡牌与地图数据
- 图片与图标资源
- Node.js 服务端
- 数据库脚本
- 测试文件

`VS Code` 对这种混合型项目非常合适。

### 3. 推荐扩展

当前阶段推荐安装：

- `ESLint`
- `Prettier`
- `GitLens`
- `Error Lens`
- `Markdown All in One`
- `Vitest` 或 `Jest` 相关扩展

未来如果引入数据库，再考虑安装：

- `Prisma`

## 三、技术路线总原则

### 1. 技术栈不要一开始就做得太重

- 不要开局就引入完整游戏引擎体系
- 不要为了“像游戏”而牺牲开发清晰度
- 不要让表现层先于规则层

### 2. 技术演进应分阶段进行

- 第一阶段先做规则原型
- 第二阶段再增强棋盘和动画表现
- 后端和联机能力在需要时再补上

### 3. 规则核心必须长期独立

- 最重要的不是 React 页面
- 最重要的是“规则核心可独立运行”

这意味着核心规则不能绑定在某个 UI 框架上。

## 四、第一阶段：规则原型

### 1. 阶段目标

第一阶段只做“能玩、能验证规则、能快速迭代”的最小版本。

这个阶段甚至不需要 Phaser。

普通 React 页面已经足够实现电子桌游原型。

### 2. 推荐技术栈

- `React`
- `TypeScript`
- `Vite`
- `Zustand`
- `CSS Modules` 或 `Tailwind CSS`
- `Vitest`

### 3. 第一阶段需要完成的能力

- 创建房间或开始单机游戏
- 玩家、职业、种族和隐藏身份
- 回合流程
- 抽牌、出牌和资源变化
- 昼夜切换
- 地图移动
- 事件结算
- 胜利条件判断

### 4. 为什么第一阶段不急着引入 Phaser

- 当前最重要的是验证规则是否成立
- 不是验证动画是否华丽
- 如果在规则还不稳定时就引入较重的地图渲染层，会拖慢设计迭代

### 5. 第一阶段的交付目标

- 有一个可交互的网页原型
- 可以跑完一局最小流程
- 可以验证核心循环是否有趣
- 可以暴露规则冲突和设计漏洞

## 五、第二阶段：棋盘和视觉表现

### 1. 何时进入第二阶段

当地图移动、棋子动画、范围提示、区域表现和裂变动画开始变复杂时，再进入这一阶段。

### 2. 推荐技术栈

- `React`
- `TypeScript`
- `Phaser`
- `Zustand`

### 3. React 与 Phaser 的职责分工

推荐采用以下分工：

`React`

- 菜单
- 玩家面板
- 卡牌详情
- 商店
- 聊天和日志
- 设置页面

`Phaser`

- 大地图
- 棋子移动
- 地图特效
- 战斗动画
- 大地裂变动画

### 4. 这样拆分的原因

- 不要用 `Phaser` 管理全部按钮、卡牌页面和复杂表单 UI
- 也不要只用 `React` 去硬做越来越复杂的棋盘动画

正确做法是：

- 用 `React` 处理信息界面和业务 UI
- 用 `Phaser` 处理地图表现和动态演出

## 六、推荐的完整架构

### 1. 前端

推荐前端组合为：

- `React`
- `TypeScript`
- `Vite`
- `Phaser`
- `Zustand`
- `Vitest`

### 2. 后端

在未来需要联机、多房间或持久化时，可以采用：

- `Node.js`
- `NestJS`
- `WebSocket` 或 `Socket.IO`
- `PostgreSQL`

### 3. 共享模块

整个项目最关键的共享部分应当是：

- `TypeScript` 编写的游戏规则核心

## 七、必须独立出的规则核心

### 1. 规则核心要从一开始独立

建议项目从结构上把规则核心单独拆出来，不依赖：

- `React`
- `Phaser`
- 浏览器 DOM

### 2. 推荐目录结构

```text
packages/
├── game-core/
├── game-data/
├── web-client/
└── game-server/
```

### 3. 各目录职责

`packages/game-core/`

- 纯规则逻辑
- 状态机
- 判定流程
- 回合推进

`packages/game-data/`

- 卡牌数据
- 地图数据
- 事件数据
- 职业种族配置

`packages/web-client/`

- 前端页面
- React 组件
- 棋盘界面
- UI 状态展示

`packages/game-server/`

- 房间管理
- 联机同步
- 服务端校验
- 存档与持久化

### 4. 规则核心应承担的职责

`game-core` 应只负责这类事情：

```ts
startGame();
drawCard();
movePlayer();
resolveEvent();
enterNight();
useSkill();
checkVictory();
```

### 5. 为什么这件事非常重要

这样做以后：

- 换 UI 不用重写规则
- 做网页端不用重写规则
- 做桌面端不用重写规则
- 做移动端不用重写规则
- 做 AI 玩家也不用重写规则

这非常符合“桌游操作系统”的项目定位。

## 八、文档与代码共仓管理

### 1. GDD 应当和代码放在同一个仓库

设计文档不应游离在仓库之外。

建议直接版本化管理：

```text
docs/
├── 00-项目定位.md
├── 01-世界观.md
├── 02-核心循环.md
├── 03-回合结构.md
├── 04-地图系统.md
├── 05-角色系统.md
├── 06-装备系统.md
├── 07-战斗系统.md
├── 08-卡牌系统.md
├── 09-经济系统.md
├── 10-裂变系统.md
├── 11-模块系统.md
├── 12-数值设计.md
└── 13-前端交互.md
```

### 2. 这样做的好处

- 规则修改和代码修改能对应起来
- 所有设计历史都能被 Git 保留
- 文档不会和实现脱节
- 更容易持续把规则形式化为 TypeScript 类型和函数

## 九、当前阶段的最终建议

### 1. 技术建议

当前推荐方案是：

- `VS Code`
- `TypeScript`
- `React`

初期不引入游戏引擎，等地图与动画需求成熟后再加入 `Phaser`。

### 2. 项目执行建议

第一步不应该是堆页面，而应该先建立两个核心目录：

```text
docs/
packages/game-core/
```

### 3. 当前最合理的工作方式

- 先完成 GDD
- 再把能形式化的规则逐步转为 TypeScript 类型和纯函数
- 让设计和实现在同一个仓库中共同成长

### 4. 这条路线最符合项目目标

它既保留了前端实现的优势，也保留了后续扩展为“桌游操作系统”的可能性，同时避免了过早引入过重技术导致的复杂度失控。
