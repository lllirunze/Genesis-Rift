# 《开天辟地之大地的裂变》项目定位

## 1. 项目概述

《开天辟地之大地的裂变》是一款以多人联机为主要形态的数字桌游项目。

项目灵感来源于电视剧《爱情公寓第三季》中陆展博设计的同名桌游。剧中的桌游融合了网络游戏、传统桌游、卡牌游戏、身份推理游戏和聚会游戏等多种玩法，并通过“上帝”、身份牌、种族、职业、元宝、点卡、昼夜变化、平安夜、创界山等设定，构建出一套规模庞大、规则复杂且充满荒诞感的游戏体系。

由于电视剧主要服务于剧情和喜剧效果，剧中只展示了部分规则，许多机制并未被完整解释，部分概念之间也存在重叠、缺失或难以直接运行的问题。

本项目的目标不是简单复刻电视剧中的片段，也不是重新设计一款仅仅“风格相似”的桌游，而是在尽可能保留剧中原有规则、名称和世界观的基础上，对缺失内容进行系统化补全，使其成为一款规则完整、逻辑统一、能够实际运行的多人联机数字桌游。

---

## 2. 项目核心定位

本项目同时具有以下三层定位。

### 2.1 电视剧同人桌游

项目首先是一款基于《爱情公寓》中陆展博原创设定进行完善的同人游戏。

游戏需要保留原作中具有代表性的内容，包括但不限于：

* 游戏名称《开天辟地之大地的裂变》
* 创界山
* 上帝
* 身份牌
* 种族与职业
* 血牌
* 元宝
* 点卡
* 昼夜变化
* 黑夜行动
* 平安夜
* 杀手、法师、妖侠、魔王、神仙、长老等角色或身份概念
* 昆仑奴等剧中出现的特殊设定
* 多种桌游、网游和聚会游戏相互融合的整体风格

所有后续设计都必须建立在这些剧中内容之上，而不是将其替换为一套完全原创的奇幻桌游规则。

### 2.2 多人联机数字桌游

本项目的主要运行形态为多人联机游戏，而不是只面向单人体验的本地游戏。

玩家可以通过房间系统进入同一局游戏，在共享地图和统一规则下进行：

* 探索
* 战斗
* 交易
* 合作
* 对抗
* 隐藏身份行动
* 阵营博弈
* 资源竞争
* 剧情事件
* 创界山终局争夺

数字化的核心价值不只是将纸质棋盘搬到网页中，而是让系统主动承担复杂规则的判断和结算工作。

系统应负责：

* 发牌
* 洗牌
* 掷骰
* 回合推进
* 昼夜切换
* 隐藏信息管理
* 行动合法性检查
* 数值计算
* 状态变化
* 冲突处理
* 事件触发
* 胜负判定
* 日志记录
* 掉线恢复

玩家的主要精力应当用于决策、推理、交流和博弈，而不是手工计算大量数值。

### 2.3 桌游规则操作系统

本项目不将所有玩法强行写进一套固定流程，而是采用“桌游规则操作系统”的设计思想。

整个游戏由一套稳定的基础规则和多个可组合的玩法模块构成。

基础规则负责提供：

* 玩家
* 房间
* 回合
* 行动
* 状态
* 卡牌
* 地图
* 资源
* 事件
* 隐藏信息
* 随机数
* 胜负判定
* 日志与回放

具体玩法则通过模块接入，例如：

* 昼夜模块
* 隐藏身份模块
* 战斗模块
* 交易模块
* 赌场模块
* 地下城模块
* 狼人之夜模块
* 礼物交换模块
* 商业经营模块
* 创界山模块
* 大地裂变模块

在经典模式中，系统启用一套最接近陆展博原始设计的标准模块组合。

在自定义模式中，房主可以调整或选择部分扩展模块，从而形成不同的对局体验。

《开天辟地之大地的裂变》不是多个小游戏的简单堆叠，而是由统一底层规则承载多个玩法模块的完整游戏世界。

## 3. 项目愿景

本项目希望最终呈现出一款具有以下特征的数字桌游。

### 3.1 看起来复杂，实际可以运行

游戏应当保留陆展博式的庞大、夸张和复杂感。

玩家第一次进入游戏时，应当能够感受到：

* 规则很多
* 世界很大
* 身份很多
* 卡牌很多
* 可以做的事情很多
* 不同游戏机制似乎都被融合了进来

但是，这种复杂不能演变成不可理解的混乱。

所有机制必须拥有明确归属，所有行动必须拥有统一流程，所有规则冲突必须有稳定的处理顺序。

系统可以复杂，但玩家在某一时刻需要做出的选择必须清晰。

### 3.2 像陆展博设计的游戏

本项目不能被优化成一款过于克制、简洁和标准化的现代桌游。

游戏需要保留一种明显的“陆展博感”：

* 热衷于加入新系统
* 认为更多规则意味着更多乐趣
* 喜欢将不同游戏机制结合起来
* 对职业、装备、属性和世界观进行层层叠加
* 存在一些不一定服务于主线，但很有趣的系统
* 规则说明听起来复杂，却有内部逻辑
* 同时具有认真设计和荒诞喜剧的双重气质

游戏中可以存在钓鱼、赌博、礼物交换、股票、转职、点卡、夜间身份行动等看似跨度很大的玩法，但这些玩法必须通过地图、事件和模块系统合理接入。

### 3.3 每一局都是一个完整故事

每局游戏不应只是玩家轮流出牌并计算分数，而应当形成一段完整的世界历程。

一局标准游戏将经历：

```
开天
→ 辟地
→ 发展
→ 冲突
→ 裂变
→ 创界
```

玩家从世界初生阶段开始探索和成长，随后建立关系、获取装备、形成阵营并完成个人目标。

随着裂变逐渐发生，原本稳定的地图、资源和规则开始被破坏。玩家需要在合作、背叛、生存和争夺之间做出选择，并最终进入创界山阶段。

对局结束后，系统应当能够通过事件日志和战报还原整局游戏中发生的主要故事。

## 4. 核心玩家体验

本项目希望为玩家提供以下几类核心体验。

### 4.1 探索未知世界

玩家进入由多个区域组成的创界大陆，通过移动、调查和事件逐步了解世界。

不同区域可以拥有不同玩法规则，例如：

* 城镇负责交易和任务
* 酒馆负责情报和聚会游戏
* 赌场负责赌博和高风险奖励
* 森林负责探索和怪物遭遇
* 神殿负责身份、阵营和祝福
* 地下城负责合作战斗与宝物获取
* 创界山负责最终阶段

地图不只是空间背景，也是各种玩法模块的入口。

### 4.2 构建独特角色

玩家角色不是单一英雄牌，而是由多个维度共同组成：

* 种族
* 职业
* 身份
* 阵营
* 属性
* 技能
* 装备
* 状态
* 个人目标

同一个职业在不同身份、种族和装备组合下，可以形成完全不同的玩法。

### 4.3 与其他玩家博弈

玩家之间不仅存在直接战斗，还可以进行：

* 交易
* 谈判
* 结盟
* 欺骗
* 情报交换
* 临时合作
* 阵营协作
* 隐藏行动
* 背叛
* 共同对抗世界灾难

游戏应鼓励玩家交流，但不能要求所有信息都依赖口头承诺。

关键行动和规则效果需要由系统正式记录和裁定。

### 4.4 应对不断变化的规则

大地裂变发生后，游戏不仅改变地图，也会改变规则环境。

可能出现：

* 部分道路断裂
* 区域被摧毁
* 城镇停止服务
* 资源价格上涨
* 怪物强化
* 卡牌发生变异
* 昼夜周期异常
* 部分身份被迫公开
* 原有胜利条件发生变化

玩家需要根据世界状态调整策略，而不能始终执行固定套路。

### 4.5 争夺创界山

创界山是游戏最重要的终局舞台，但登上创界山不一定直接等于获胜。

玩家可能需要：

* 收集登山资格
* 获得神器
* 击败守卫
* 完成身份任务
* 与其他玩家合作开启山门
* 在山顶进行最终竞争
* 对抗终局敌人
* 决定新世界的规则

创界山既是地理终点，也是所有玩家目标发生碰撞的位置。

## 5. 设计原则

### 5.1 剧中明确规则优先保留

电视剧中通过台词、画面或剧情明确展示的规则，应当作为最高优先级内容保留。

除非某项规则完全无法运行，否则不应因为现代桌游设计习惯而将其删除或替换。

例如，`上帝`，`元宝`，`点卡`，`平安夜`等名称可能并不符合统一奇幻世界观，但这些内容正是原作特色的一部分，不应擅自改成普通的主持人、金币、能量值或和平阶段。

### 5.2 缺失内容允许合理补全

电视剧没有完整展示所有规则，因此项目必须进行补全。

补全规则时需要遵循以下条件：

* 不与剧中明确内容冲突
* 能解释剧中已经出现的行为
* 能够在联机环境下稳定运行
* 能与其他模块进行组合
* 保留原作的复杂和荒诞风格
* 不为了增加内容而无意义堆叠系统

合理补全的目标是让原有规则成立，而不是借用原作名称设计一款无关的新游戏。

### 5.3 原创扩展不得覆盖原作核心

允许加入电视剧中没有出现的区域、职业、卡牌、事件和玩法，但原创内容必须处于扩展地位。

原创扩展不得：

* 修改游戏的核心名称
* 删除创界山
* 弱化上帝系统
* 移除昼夜和身份行动
* 将元宝、点卡等内容彻底替换
* 将游戏改造成单纯的卡牌对战
* 将游戏改造成单纯的角色扮演游戏
* 将复杂融合风格改造成极简竞技游戏

原创内容的作用应当是连接、解释和丰富原作，而不是取代原作。

###5.4 复杂但有秩序

游戏可以拥有多个资源、多种卡牌、多层身份和大量规则，但所有复杂内容必须遵循统一结构。

每项机制都要能够回答：

* 它属于哪个系统？
* 它在什么时候触发？
* 它需要满足什么条件？
* 它会改变哪些状态？
* 它与其他规则冲突时如何处理？
* 它是否对所有玩家可见？
* 它如何被系统记录？

无法回答这些问题的机制不能直接加入正式规则。

### 5.5 自动判定优先

数字化项目应尽量减少玩家手工处理规则。

系统应自动完成：

* 数值计算
* 卡牌条件判断
* 技能触发
* Buff与Debuff结算
* 身份信息分发
* 随机结果生成
* 行动顺序处理
* 裂变推进
* 胜负检查

复杂计算可以存在于后台，但前端需要向玩家解释最终结果。

例如：
```
基础伤害：5
武器加成：+2
火焰易伤：+2
护甲减免：-1
最终伤害：8
```

玩家既不需要手算，也能够理解结果来源。

### 5.6 服务端拥有最终裁定权

多人联机环境中，客户端只负责展示状态和提交玩家意图。

所有关键规则必须由统一的权威系统处理，包括：

* 当前回合
* 手牌内容
* 隐藏身份
* 骰子结果
* 洗牌顺序
* 行动是否合法
* 技能是否生效
* 伤害结果
* 交易结果
* 胜负判定

任何客户端都不能直接决定或修改最终游戏状态。

### 5.7 所有状态变化可追溯

游戏中的重要变化都需要被记录。

例如：

* 玩家为什么失去生命
* 玩家为什么获得元宝
* 某张卡为什么无法使用
* 某个身份为什么被揭示
* 某个区域为什么发生裂变
* 某场战斗为什么由某一方获胜

系统应保留结构化日志，为以下功能提供基础：

* 战报
* 回放
* 掉线恢复
* 观战
* 规则解释
* 问题排查
* 数值平衡分析

### 5.8 联机体验优先于纸面模拟

部分剧中规则如果直接按照纸质桌游方式实现，可能会产生长时间等待、信息泄露或大量手工操作。

在不改变核心意义的前提下，可以对流程进行数字化改造。

例如：

* 上帝由系统承担
* 夜间行动由玩家分别秘密提交
* 同时行动由系统统一结算
* 无需玩家手动闭眼
* 隐藏卡牌由权限系统控制
* 复杂效果由系统自动处理

数字版应保留原规则的体验，而不必机械复制纸质操作形式。

### 5.9 模块之间应低耦合

每个玩法模块应尽量通过统一的行动、事件和状态接口与其他系统协作。

例如，赌场模块不应直接修改角色底层代码，而应通过：

* 扣除货币
* 生成随机事件
* 发放奖励
* 添加状态
* 记录日志

等通用能力完成自身逻辑。

这样才能使不同模块自由组合，同时避免新增一个小游戏就破坏整套系统。

### 5.10 先完成可玩的核心，再扩展内容

陆展博式设计容易无限加入新系统，因此项目必须控制开发顺序。

第一阶段不追求完整实现所有剧中提到的游戏种类，而是先保证最小可玩版本成立。

最小可玩版本应至少包含：

* 多人房间
* 基础地图
* 上帝与世界牌
* 昼夜循环
* 基础角色
* 种族、职业和身份
* 移动
* 战斗
* 卡牌
* 交易
* 事件
* 大地裂变
* 创界山终局
* 胜负判定

在核心循环稳定后，再加入赌场、商业、聚会游戏、地下城和其他扩展模块。

## 6. 原作规则保护机制

为了避免后续设计逐渐偏离电视剧原设定，所有规则文档统一使用以下三种标记。

【剧中明确】

表示该内容在电视剧中通过台词、画面或剧情被直接确认。

此类内容具有最高保留优先级。

示例：

```
【剧中明确】
游戏中存在“上帝”，由上帝负责发牌、翻牌和推进部分流程。
```

【合理补全】

表示电视剧已经提供了概念，但缺少完整执行方式，需要为保证游戏可运行而补充。

示例：
```
【合理补全】
数字版中的上帝默认由服务端规则系统承担，不占用普通玩家席位。
```

【原创扩展】

表示该内容并未在电视剧中明确出现，是为了增强完整性、可玩性或联机体验而新增。

示例：
```
【原创扩展】
自定义房间允许一名玩家担任真人上帝，并获得额外事件控制权限。
```

所有核心机制在进入正式设计前，都应明确其属于哪一类。

## 7. 游戏模式定位

项目计划支持多种游戏模式，但经典模式始终是最重要的标准模式。

### 7.1 经典模式（暂时只考虑这个模式即可）

经典模式以还原陆展博设计理念为目标。

该模式默认启用：

* 上帝系统
* 世界牌
* 昼夜循环
* 身份系统
* 种族与职业
* 血牌
* 元宝
* 点卡
* 地图探索
* 战斗
* 交易
* 平安夜
* 大地裂变
* 创界山终局

经典模式中的核心模块不能被房主关闭。

### 7.2 自定义模式（暂不开发）

自定义模式允许房主调整部分非核心规则，例如：

* 地图规模
* 裂变速度
* 玩家人数
* 回合时间
* 扩展模块
* 卡牌池
* 职业池
* 身份池
* 终局难度

自定义模式仍需遵循底层规则操作系统，不允许产生无法判定的规则组合。

### 7.3 新手模式（暂不开发）

新手模式降低初始复杂度，但不改变核心世界观。

可以采用：

* 减少角色维度
* 限制卡牌种类
* 关闭部分扩展模块
* 提供行动推荐
* 自动跳过无效响应窗口
* 增加规则提示

新手模式是经典模式的简化入口，而不是另一款游戏。

### 7.4 剧情模式（暂不开发）

剧情模式用于呈现特定世界事件或预设对局。

玩家可以体验：

* 固定地图
* 指定角色
* 特定任务
* 预设裂变过程
* 特殊Boss
* 剧中情节致敬

剧情模式可以兼顾单人或多人合作，但不作为项目最主要的玩法定位。

## 8. 目标用户

本项目主要面向以下用户。

### 8.1 《爱情公寓》观众

这类玩家对陆展博和《开天辟地之大地的裂变》具有情感记忆，希望看到剧中只展示了部分内容的桌游被真正实现。

### 8.2 桌游爱好者

这类玩家对隐藏身份、卡牌、地图探索、资源管理、阵营博弈和模块化规则感兴趣。

### 8.3 角色扮演与策略游戏玩家

这类玩家希望通过职业、装备、技能和个人目标构建角色，并在不断变化的世界中制定长期策略。

### 8.4 喜欢多人社交游戏的玩家

这类玩家更关注谈判、欺骗、合作、背叛和玩家之间产生的故事。

## 9. 项目边界

为了避免项目失去重点，当前阶段明确以下边界。

### 9.1 当前阶段重点

当前优先完成：

* 完整游戏设计文档
* 剧中规则整理
* 世界观统一
* 核心循环
* 回合结构
* 角色系统
* 战斗系统
* 卡牌系统
* 经济系统
* 裂变系统
* 模块系统
* 联机交互原则

### 9.2 当前阶段暂不确定

以下内容暂不急于决定：

* 最终前端框架
* 最终后端框架
* 数据库类型
* 部署平台
* 美术风格细节
* 商业化方式
* 移动端实现
* 语音聊天
* 匹配与排位系统

这些内容可以在核心规则稳定后再进行技术选型。

### 9.3 当前阶段不追求

项目初期不追求：

* 一次性实现所有扩展模块
* 制作数百张卡牌
* 完成大型开放世界
* 制作复杂三维画面
* 建立完整电竞体系
* 完成商业发行标准

项目首先要证明核心规则能够成立，并能够支撑一局完整的多人游戏。

## 10. 最小可玩版本定位

最小可玩版本暂定支持3至4名玩家。

一局游戏应包含：
```
创建房间
→ 玩家准备
→ 分配角色、种族、职业和身份
→ 创建地图
→ 上帝翻世界牌
→ 玩家执行昼夜行动
→ 探索、战斗、交易和成长
→ 世界裂变逐步发生
→ 创界山开放
→ 玩家进入终局
→ 系统判定胜负
→ 生成战报
```

最小可玩版本应具备以下内容：

* 1套基础地图
* 4种基础种族
* 4种基础职业
* 4至6种隐藏身份
* 基础属性与资源
* 世界牌
* 行动牌
* 装备牌
* 事件牌
* 基础怪物
* 昼夜循环
* 平安夜
* 玩家交易
* 基础战斗
* 裂变等级
* 创界山终局
* 至少两类胜利方式

最小可玩版本的目标不是展示内容数量，而是验证：

* 多层角色体系是否可理解
* 模块能否协同运行
* 昼夜和身份规则是否适合联机
* 裂变是否能有效推动终局
* 玩家之间是否能够产生有趣互动
* 系统是否能自动处理复杂规则

## 11. 项目成功标准

项目是否成功，不以规则数量或卡牌数量作为唯一判断依据。

核心成功标准包括：

### 11.1 原作辨识度

玩家进入游戏后，能够明确感受到这是陆展博的《开天辟地之大地的裂变》，而不是套用了名称的普通奇幻桌游。

### 11.2 完整可玩性

游戏能够从开局持续运行至终局，不依赖设计者临时解释规则。

### 11.3 规则可理解性

玩家能够知道自己当前可以做什么、为什么能做、行动之后发生了什么。

### 11.4 联机稳定性

隐藏信息、同时行动、超时、掉线和重新连接都有明确规则。

###11.5 模块扩展性

新增模块时，不需要推翻核心回合、状态和行动系统。

### 11.6 对局故事性

一局结束后，玩家能够回忆出其中发生过的关键事件、合作、背叛和转折，而不只是最终分数。

### 11.7 裂变终局有效性

大地裂变能够打破僵局、压缩地图、改变策略，并将游戏稳定推进到创界山终局。

---

## 12. 项目最高原则

本项目后续所有设计必须遵守以下三条最高原则：

1. 剧中明确规则优先保留。
2. 剧中缺失规则允许进行合理补全。
3. 原创扩展不得破坏陆展博式的荒诞、复杂与融合感。

当可玩性、现代桌游设计习惯与原作规则发生冲突时，应优先尝试通过自动结算、界面引导、规则分层和模块化设计解决问题，而不是直接删除原作特色。

本项目最终希望完成的，不是一款借用了电视剧名称的桌游，而是一套能够让玩家真正相信“这就是陆展博当年设计出来的那款游戏”的完整数字化实现。
