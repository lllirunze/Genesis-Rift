# 《开天辟地之大地的裂变》数值系统总览

## 1. 文档目的

数值系统（Attribute System）是整个游戏的底层支撑系统，负责管理角色的各项能力、成长、战斗、探索以及其他玩法所涉及的全部数值计算。

本系统遵循统一计算、数据驱动、配置与运行时解耦的设计原则，使所有数值均采用一致的计算方式，并能够在后期方便地进行平衡调整、功能扩展与维护。

本章节主要介绍数值系统的整体架构，各模块的详细设计将在后续文档中分别说明。

---

## 2. 设计目标

数值系统遵循以下设计目标。

### 2.1 统一计算模型

所有属性均采用统一的计算方式，不针对某一项属性设计特殊逻辑。

无论是生命值、攻击力、移动力、暴击率还是视野范围，都遵循相同的数据结构与计算流程，从而保证系统的一致性。

### 2.2 数据驱动（Data Driven）

游戏中的固定数值全部由配置文件维护，而不是写入程序逻辑。

包括但不限于：

* 各派生属性计算系数
* 静态偏移量（Static Offset）
* 默认值
* 边界值
* 取整规则

因此，当需要调整游戏平衡时，仅需修改配置文件即可，无需修改程序代码。

### 2.3 配置与运行时解耦

数值系统将固定规则与角色实时状态彻底分离。

系统固定规则负责定义：

* 如何计算属性
* 属性之间的关系
* 固定偏移量
* 默认规则

角色自身只负责维护：

* 当前基础属性
* 当前受到的动态修饰

这样能够有效降低模块之间的耦合程度，提高维护效率。

### 2.4 易扩展

新增任意派生属性时，不需要修改计算逻辑。

只需新增一条属性配置，即可自动参与整个计算流程。

例如后续新增：

* 幸运值
* 感知
* 制造成功率
* 护甲穿透
* 法术穿透
* 侦察范围
* 资源采集效率

均无需修改底层计算框架。

---

## 3. 数值系统组成

整个数值系统由以下五部分组成：

```text
数值系统
│
├── 基础属性（Primary Attributes）
├── 派生属性（Derived Attributes）
├── 属性修饰器（Attribute Modifiers）
├── 属性计算与结算顺序
└── 整数与边界规则
```

各模块职责如下：

| 模块 | 职责 |
| --- | --- |
| 基础属性 | 定义角色最基础的能力，是所有派生属性的计算来源。 |
| 派生属性 | 由基础属性计算得到，决定角色实际表现能力。 |
| 属性修饰器 | 管理所有静态偏移量与动态偏移量。 |
| 属性计算与结算顺序 | 定义整个数值系统统一的计算流程。 |
| 整数与边界规则 | 统一所有属性的取整方式、上下限及异常处理规则。 |

---

## 4. 基础属性

基础属性（Primary Attributes）是角色最基础的能力值。

本项目采用五维基础属性设计：

* 力量（Strength）
* 体质（Constitution）
* 灵力（Spirit）
* 敏捷（Agility）
* 悟性（Insight）

角色创建完成后，将根据初始职业、种族以及成长情况得到角色的当前基础属性。

基础属性本身不直接体现游戏效果，而是作为所有派生属性计算的唯一输入。

---

## 5. 派生属性

派生属性（Derived Attributes）是角色最终参与游戏玩法的实际能力。

例如：

* 最大生命值
* 最大灵力值
* 物理攻击
* 法术攻击
* 护甲
* 法术抗性
* 命中率
* 暴击率
* 移动力
* 视野
* 幸运值

所有派生属性均由统一公式计算，不允许存在特殊计算逻辑。

每一项派生属性均拥有独立的配置项，可单独调整其计算方式。

---

## 6. 属性修饰器

为了保证系统统一，所有属性修正均采用 Offset（偏移量）的设计方式。

根据作用阶段，可分为：

* 基础属性偏移量（Primary Offset）
* 派生属性偏移量（Derived Offset）

根据生命周期，可分为：

* Static Offset（静态偏移量）
* Dynamic Offset（动态偏移量）

因此，整个系统统一存在四种偏移量：

| 名称 | 说明 |
| --- | --- |
| `primaryStaticOffset` | 基础属性静态偏移量，由系统固定配置提供。 |
| `primaryDynamicOffset` | 基础属性动态偏移量，由装备、Buff、天气等运行时产生。 |
| `derivedStaticOffset` | 派生属性静态偏移量，由系统固定配置提供。 |
| `derivedDynamicOffset` | 派生属性动态偏移量，由装备、Buff、技能等运行时产生。 |

这种划分能够统一所有属性修饰方式，避免出现大量特殊判断。

---

## 7. 数据驱动设计

整个数值系统采用配置驱动（Configuration Driven）的设计思想。

固定规则由系统资源文件维护，例如：

* 属性计算系数（Coefficients）
* `primaryStaticOffset`
* `derivedStaticOffset`

运行时数据由角色实例维护，例如：

* 当前基础属性（Current Primary Attributes）
* `primaryDynamicOffset`
* `derivedDynamicOffset`

系统根据配置与角色实时状态共同计算最终数值，而不会将固定规则写入角色对象。

---

## 8. 设计原则

整个数值系统遵循以下设计原则：

### 8.1 原则一：统一计算

所有属性均采用统一的数据结构与计算流程，不为单个属性设计特殊算法。

### 8.2 原则二：配置优先

所有固定规则均由配置文件维护，程序只负责读取配置并完成计算。

### 8.3 原则三：运行时解耦

角色仅保存自身状态，不保存系统规则；系统规则也不直接依赖角色对象。

### 8.4 原则四：易于扩展

新增属性无需修改底层代码，仅需新增配置即可参与整个数值系统。
